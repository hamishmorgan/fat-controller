# Icon Sync Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Fully implement service icon fetching, diffing, applying, and rendering so that `icon` in a config file is synced to Railway like any other field.

**Architecture:** Icon is a project-scoped property on the Railway Service object (not per-environment like ServiceInstance fields). It is updated via the already-generated `ServiceUpdate` mutation with `ServiceUpdateInput{Icon: *string}`. The work follows the standard pipeline: fetch â†’ diff â†’ apply â†’ render.

**Tech Stack:** Go, GraphQL (genqlient generated), TOML (BurntSushi/toml), existing `diff.Change` / `apply.Applier` patterns.

## Notes on icon semantics

- **Empty desired icon = no-op**: If a user doesn't write `icon = "..."` in their config, `desired.Icon` is `""` and the diff step skips it (Task 2, Step 4). This avoids accidentally clearing icons.
- **Icon is project-scoped**: `UpdateServiceIcon` calls `ServiceUpdate` (not `ServiceInstanceUpdate`). This means applying a config with `icon = "server"` changes the icon in all environments, which is consistent with Railway's data model.
- **Adopt captures icon**: After Task 4, `FetchLiveConfig` returns `ServiceConfig.Icon` populated (Task 1), and `liveToTOMLServices` writes it to the config file (Task 4). So `adopt` will save whatever icon Railway has.
